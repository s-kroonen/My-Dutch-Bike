## What's done

1. **Repo + Unity project skeleton** — Unity 6 (URP), merge-friendly settings (Force Text,
   visible meta files), Git + Git LFS, GitHub Actions-free CI-ready structure per
   `docs/REPO_SETUP_GUIDE.md`.
2. **Plan expansion (ADR-0005)** — the docs now cover **two bike frame archetypes** (city,
   race) and **two riding environments** (city street loop, race track loop) as core v1.0
   scope, generalized on the same part/fastener system rather than being a single bike.
3. **Part/fastener system** (`Assets/Code/Parts/`) — `PartDefinition`/`SocketDefinition`/
   `FastenerSlot` (authored ScriptableObjects) + `AssemblyState`/`PartState`/`FastenerState`
   (plain-data runtime state, matches `docs/PART_SYSTEM_DESIGN.md`'s save-payload design).
4. **12-part bike BOM** (2 frames + 10 shared parts) authored as placeholder primitive
   prefabs + data assets via `Assets/Editor/BikeBomAuthoring.cs` (idempotent).
5. **`BikeAssembly` runtime** (`Assets/Code/Bike/`) — spawns the frame, tracks install
   order/dependencies, fastener tightness, and whether the bike is complete enough to ride.
6. **Interaction** (`Assets/Code/Interaction/PlayerInteractor.cs`) — raycast-based pickup,
   place-on-socket, tighten/loosen (LMB/RMB), mount/dismount, crosshair + contextual prompt
   text, cursor lock (Escape to free it).
7. **Visual debug aids** — sockets/fasteners are small visible spheres (not just invisible
   trigger colliders): sockets glow green until occupied, fasteners lerp red→green with
   tightness. F1 toggles a full state readout (`BikeDebugOverlay`).
8. **Two greybox scenes** (`Assets/Scenes/City.unity`, `RaceTrack.unity`), each generated by
   `Assets/Editor/GreyboxSceneAuthoring.cs`: ground/road, perimeter markers, a bike stand
   (with a placeholder static repair-stand prop) holding a fully-assembled-state frame, all
   other parts scattered loose nearby with `Rigidbody`s (fall/settle under gravity), and a
   player rig (`FirstPersonController` + `PlayerInteractor` + camera).
9. **Per-archetype handling data** (`Assets/Data/Mechanics/*.asset`, `FrameMechanics`) —
   city is slower/more stable, race is faster/twitchier (`BikeRideController`).
10. **Verified twice headlessly** via `Assets/Editor/M1SmokeTest.cs` (programmatically
    installs every part + tightens every fastener on both scenes, asserts completion):
    once on Unity 6000.0.41f1, again after the user upgraded to **6000.5.4f1** — both
    `ALL PASS`, no regressions from the upgrade.
11. **Repo-level Claude Code config** — `.claude/settings.json` + `.claude/hooks/`:
    - Blocks Unity batchmode launches while the Editor is already open (this bit the
      session twice before the hook existed).
    - Blocks any Bash command containing a `Co-Authored-By: Claude` trailer (repo
      convention, see `CLAUDE.md`).

## What's confirmed by an actual human playtest

- Project opens and runs in the Editor.
- Picking up a loose part works.

**Everything else in the interaction loop (placing parts on sockets, tightening bolts,
mounting/riding, the crosshair/prompt legibility, the debug overlay, city-vs-race handling
feel) has only been verified by the scripted smoke test, not by a human clicking through it.**
The smoke test proves the *logic* is internally consistent (install order, dependencies,
tightness thresholds) — it says nothing about whether it *feels* right, whether bolt hitboxes
are a reasonable size to aim at, or whether anything looks broken visually.

## Known rough edges / not yet done

- **Physics tuning untested.** Loose parts now have `Rigidbody`s and fall under gravity
  (per user request) — cylindrical parts (wheels/tires/crankset) use Unity's default
  capsule-collider approximation for a `PrimitiveType.Cylinder` and may roll further than
  intended before settling. Nobody has watched this happen yet.
- **Cosmetic material collision:** `City.unity`'s and `RaceTrack.unity`'s ground/road
  materials share the same asset names (`GroundMat`, `RoadMat`) in
  `GreyboxSceneAuthoring.cs`, so the two scenes' intended slightly-different tints both end
  up as whichever was authored last. Harmless, not fixed, low priority.
- **No save/load system** — flagged in the plan as the highest-risk system (§6.3), not
  started. `AssemblyState` is designed to serialize cleanly to JSON when this happens.
- **No wear/condition decay** — `PartState.condition` field exists but nothing decrements it
  over use/time yet (plan §4.7, §6.3).
- **No PSX render pipeline** — `docs/ART_DIRECTION_SPEC.md` calls for vertex snapping,
  affine texture mapping, dithering, etc. None of that exists yet; everything renders with
  plain URP defaults. Intentional — this milestone is about mechanics, not style.
- **Legacy Input Manager**, not the new Input System package — deliberately deferred (the
  package isn't bundled locally on this machine, would need a network fetch). Fine for
  current WASD/mouse/E/Q/LMB/RMB needs; revisit if input gets more complex (rebinding, etc).
- **Team/ownership** (plan §9) still parked/open — solo project so far.

## Machine-specific things NOT to assume carry over

- This machine had `commit.gpgsign=true` with SSH-format signing and needed the Windows
  `ssh-agent` **service** (not a per-shell `ssh-agent`) running for commits to succeed
  without hanging. A different machine may have different (or no) signing config — don't
  assume the same fix applies; diagnose fresh if `git commit` hangs (see `CLAUDE.md`).
- `gh` (GitHub CLI) was not installed on this machine; the GitHub remote was wired up via a
  URL the user pasted after creating the repo manually. A different machine might have
  `gh` available — fine to use it if so.
- Local Claude Code memory (`~/.claude/projects/.../memory/`) on this machine has a
  `feedback_no_claude_coauthor.md` entry — **that memory does not travel to another
  machine.** This is exactly why the no-co-author rule now also lives in `CLAUDE.md` and is
  enforced by a repo-committed hook: don't rely on memory for anything that matters beyond
  one machine, put it in the repo instead.
